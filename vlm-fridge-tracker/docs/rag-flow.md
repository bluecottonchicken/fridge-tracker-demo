# RAG 检索流程详解

## 概述

本项目采用 **Context Injection（上下文注入）** 模式，而非传统的 query-based retrieval 模式。具体来说，不是根据当前视频帧内容动态检索相关文档，而是在每次 VLM 分析前，将用户历史数据（冰箱状态、高频物品、修正记录、品类库）结构化注入 prompt，为 VLM 提供先验知识。

整体为**结构化上下文注入 + 后置品类匹配**的两阶段设计。

---

## 阶段一：构建 RAG 上下文（VLM 分析前）

在调用 Gemini 分析图片之前，从数据库中检索四类信息，拼装为文本注入 VLM prompt。

### 1.1 检索模块

| 模块 | 文件 | 数据来源 | 检索方式 | 输出 |
|------|------|---------|---------|------|
| 冰箱当前状态 | `rag/state_tracker.py` | `inventory_events` 表 | SQL 聚合全部 put_in/take_out 事件 | 当前冰箱内物品名称+数量 `{名称: 数量}` |
| 用户高频物品 | `rag/user_history.py` → `get_frequent_items` | `item_knowledge` 表 | SQL 按 user_id 查询，Python 端按物品名分组计数，优先取纠错后描述 | top 10 物品名+描述+频次 |
| 历史修正映射 | `rag/user_history.py` → `get_alias_mappings` | `item_knowledge` 表 | SQL 筛选 `source="user_corrected"`，按正确物品名聚合 | 聚合修正记录：正确名+所有错误名+修正次数+外观描述 |
| 已知品类列表 | `rag/cross_user_matcher.py` | `item_categories` 表 | SQL 直接取全部品类（上限 30） | 品类名+外观描述 |

### 1.2 上下文拼装

由 `rag/prompt_enhancer.py` → `build_context()` 将四类信息拼装为结构化文本：

```
【用户上下文】
上次关门后冰箱内有：牛奶 x2、鸡蛋、西红柿

该用户常见物品：
  - 伊利纯牛奶（白色利乐砖）
  - 鸡蛋（散装）

历史修正记录（以下物品曾被错误识别，请格外注意避免重复犯错）：
  - 「培根」已被修正3次，请务必注意！曾被错误识别为："无糖可乐"、"猪肉"、"火腿肠"
    正确外观特征：扁平长方形透明塑料盒，红白相间肉片
  - 曾将 "白色纸盒" 错误识别为「伊利纯牛奶」（白色利乐砖包装）

已知食物类别（请用这些类别名称识别物品，而非具体品牌名）：
  - 牛奶（白色纸盒/利乐砖）
  - 鸡蛋（散装/盒装）
  ...
```

### 1.3 注入位置

上下文文本被注入到 `vlm/prompt_template.py` → `build_user_prompt()` 的最前面，位于图片分析指令之前。VLM 在分析图片时可以参考这些已知信息，优先使用已知名称识别物品。

二次识别（`build_refine_prompt`）同样会注入 RAG 上下文，帮助低置信度物品的重新识别。

## 阶段二：品类匹配（VLM 分析后）

VLM 识别完物品、用户确认/修正后，进行品类归类。这个阶段使用了**向量相似度检索**。

### 2.1 匹配流程

位于 `rag/category_matcher.py` → `match_category()`：

```
输入：物品名称 + 描述 + 预计算的 embedding
         │
         ▼
┌─ Step 1: Embedding 相似度匹配 ─────────────────────┐
│  将物品 embedding 与 item_categories 表中所有品类的   │
│  embedding 批量计算余弦相似度（numpy 向量化）          │
│  阈值 ≥ 0.65 → 命中，归入该品类                     │
└─────────────────────────────────────────────────────┘
         │ 未命中
         ▼
┌─ Step 2: Gemini 文本推理 ──────────────────────────┐
│  将物品名+描述+现有品类列表发给 Gemini               │
│  Gemini 判断属于哪个现有品类，或建议新建              │
└─────────────────────────────────────────────────────┘
         │
         ▼
┌─ Step 3: 结果处理 ─────────────────────────────────┐
│  匹配已有品类 → 更新品类的 user_names 列表           │
│  新建品类 → 生成 embedding，写入 item_categories 表  │
└─────────────────────────────────────────────────────┘
```

### 2.2 Embedding 技术细节

- **模型**：`text-embedding-004`（Gemini embedding API）
- **存储**：embedding 向量序列化为 JSON 字符串，存储在 SQLite TEXT 列中
- **计算**：使用 numpy 批量矩阵运算（`batch_cosine_similarity`），一次计算查询向量与所有品类的相似度
- **批量生成**：一次分析会话中的所有物品共享一次 embedding API 调用（`get_embeddings_batch`）

### 2.3 品类库初始化

首次运行时（`rag/category_init.py`），调用 Gemini 生成 50-80 个常见食物品类及其外观描述，批量生成 embedding 并写入 `item_categories` 表。后续新物品匹配不到时自动扩展品类库。

## 数据流全景

```
视频帧 → 门状态检测 → 片段切割 → 关键帧提取
                                      │
                        ┌─────────────┤
                        ▼             │
              ┌─ RAG 上下文构建 ─┐    │
              │ SQL 查询四类信息  │    │

              └───────┬─────────┘    │
                      │              │
                      ▼              ▼
              ┌─ Gemini VLM 分析 ────────┐
              │ 关键帧图片 + RAG 上下文    │

                      │
```

## 修正记录的聚合加权机制

历史修正记录不是简单的"原始名 → 修正名"一对一映射，而是**按正确物品名聚合**的加权结构。

### 聚合逻辑

`rag/user_history.py` → `get_alias_mappings()` 将同一物品的所有修正记录聚合为一条：

```
item_knowledge 表中的多条记录：
  source=user_corrected, original_name="无糖可乐", item_name="培根", description="..."
  source=user_corrected, original_name="猪肉",     item_name="培根", description="..."
  source=user_corrected, original_name="火腿肠",   item_name="培根", description="..."

聚合为：
  {
    "corrected": "培根",
    "wrong_names": ["无糖可乐", "猪肉", "火腿肠"],
    "correction_count": 3,
    "description": "扁平长方形透明塑料盒，红白相间肉片"
  }
```

### 分级注入策略

`rag/prompt_enhancer.py` → `build_context()` 根据修正次数分级注入：

- **修正 ≥ 2 次**：加重警告语气 + 附带外观特征描述，引导 VLM 格外注意
- **修正 1 次**：简短提示 + 可选外观描述

修正次数越多的物品排序越靠前，VLM 在 prompt 中越早看到，注意力权重越高。

### 描述来源

外观描述来自用户纠错后 Gemini 的重新描述（`gemini_client.redescribe_item()`），存储在 `item_knowledge.description` 字段。高频物品模块（`get_frequent_items`）同样优先使用纠错后的描述，确保 VLM 看到的是经过用户验证的准确描述。

## 知识库去重与上限机制

`item_knowledge` 表用于积累物品知识，同一物品可能从不同角度/场景产生多条不同描述。为防止长期膨胀，采用**去重 + 上限封顶**策略。

### 去重规则

`storage/inventory.py` → `save_to_knowledge()` 写入前检查：

1. **描述完全相同** → 跳过，不重复写入。若新记录是 `user_corrected` 而已有记录是 `vlm_accepted`，则升级 source 优先级
2. **描述不同** → 正常写入（保留不同角度的描述信息）

### 上限封顶

同一 `user_id + item_name` 最多保留 **5 条**记录（`MAX_KNOWLEDGE_PER_ITEM = 5`）。超出时按以下优先级淘汰：

1. 优先淘汰最旧的 `vlm_accepted` 记录
2. `user_corrected` 记录优先保留（用户修正的信息价值更高）
3. 若 `vlm_accepted` 不够淘汰，则从最旧的记录中补齐

### 设计目的

- 保留同一物品从不同角度拍摄的多样描述（最多 5 条）
- 用户纠错产生的描述优先级最高，不会被轻易淘汰
- 避免跑 100 次产生 100 条重复记录

## 与传统 RAG 的区别

本项目借鉴了 RAG 的核心思想（检索外部知识增强生成），但实现方式是 **Context Injection** 而非 query-based retrieval：

| 维度 | 传统 RAG (Query-based Retrieval) | 本项目 (Context Injection) |
|------|---------|--------|
| 触发方式 | 根据用户 query 动态检索 | 每次分析前固定注入用户历史上下文 |
| 检索对象 | 非结构化文档片段 | 结构化数据库记录 |
| 检索方式 | 向量相似度检索 | SQL 查询（阶段一）+ 向量匹配（阶段二） |
| 检索时机 | 生成前检索 | 生成前（上下文注入）+ 生成后（品类匹配） |
| 增强方式 | 检索结果拼入 prompt | 结构化上下文注入 prompt |
| 知识积累 | 通常静态 | 每次运行后动态积累用户修正和物品知识 |


## Notes
为什么不使用传统多模态（图片-描述对应）rag？

冰箱内摄像头角度固定且视野受限 — 物品经常被手遮挡、被其他物品挡住、只露出侧面或底部。同一瓶牛奶，正面放和侧面放在图片特征上差异巨大。

物品入场时的视觉状态极不稳定 — 用户手持物品放入/取出时，物品在运动中、有运动模糊、光照变化大（冰箱灯 + 外部光线交替），这些帧提取的图片特征噪声很高。

同品类外观高度相似 — 两盒不同品牌的牛奶在冰箱视角下几乎一样（都是白色纸盒），图片特征很难区分。但文本维度上"伊利"和"蒙牛"是完全不同的。

当前的文本 embedding + RAG 上下文注入已经够用 — 真正帮助识别准确率的是"这个用户上次放了什么"和"之前把白色纸盒修正为伊利纯牛奶"这类历史知识，这些用文本检索就能很好地解决。

臣以为，当前的方案（让 VLM 直接看图识别 + 用文本 RAG 提供上下文引导）已经是针对这个场景最合理的架构。把精力花在图片特征检索上，投入产出比不高。
