# RAG 检索流程详解

## 概述

本项目采用 **两阶段独立验证 + 结构化上下文注入 + 后置品类匹配** 的三层设计。

- **Stage 1（盲猜）**：VLM 不注入任何 RAG 上下文，独立识别物品
- **Stage 2（定向验证）**：针对每个物品，注入精准筛选的 RAG 上下文进行二次确认
- **后置品类匹配**：识别完成后，将物品归入品类体系

两阶段独立判断的核心价值在于：Stage 1 不受先验偏见影响，Stage 2 基于视觉描述（不含 Stage 1 猜测名）独立验证，两者的一致性本身就是可信度信号。

---

## Stage 1：盲猜分析（无 RAG）

### 1.1 流程

关键帧直接发送给 Gemini，不注入任何用户历史或品类信息。VLM 完全基于视觉内容独立判断。

```
关键帧图片 → Gemini VLM（无上下文）→ 原始识别结果
```

### 1.2 设计意图

- 获取不受先验偏见影响的独立信号
- 与 Stage 2 形成交叉验证（两个独立信号一致 → 可信度高）
- 避免上下文污染（如高频物品导致模型先入为主）

### 1.3 输出

```json
{
  "hand_observations": [...],
  "events": [
    {"action": "take_out", "item": "牛奶", "confidence": 0.8, "description": "白色利乐砖"}
  ],
  "fridge_state_after": [...],
  "notes": "..."
}
```

---

## Stage 2：定向 RAG 二次确认

对 Stage 1 输出的每个物品，独立构建专属 RAG 上下文进行二次验证。

### 2.1 关键帧筛选

`main.py` → `_pick_item_frames()` 从全量关键帧中挑选与目标物品相关的帧：

1. **精确匹配**：`hand_observations` 中 `holding_item` 与物品名匹配的帧
2. **Fallback**：匹配不到时，取所有手持物品的帧
3. **兜底**：首、中、尾帧

筛选出的关键帧同时保存到 `output/keyframes_stage2/segment_{N}/` 供调试对比。

### 2.2 知识库检索

`rag/knowledge_retriever.py` → `retrieve_relevant_knowledge()`：

- 基于物品 embedding 向量从 `item_knowledge` 表中召回最相似的历史经验
- 按物品名去重，确保多样性
- 返回物品名、原始名、描述、来源（`source`）、相似度

### 2.3 历史修正记录精准匹配

在主线程中为每个物品进行两层 alias 匹配：

1. **第 1 层**：Stage 1 猜测名直接命中 alias 的 `wrong_names`（精确字符串匹配）
2. **第 2 层**：知识检索结果的物品名命中 alias 的 `corrected`（跨语言/同义词桥接）

命中时，用 alias 的正确名做补充检索，打破 Stage 1 错误锚定。

### 2.4 动态上下文构建

每个物品独立构建专属 Context（`_refine_task` 内部）：

| 上下文来源 | 注入条件 | 说明 |
|-----------|---------|------|
| 冰箱当前状态 | 仅 take_out 事件 | 放入时不需参考现有状态 |
| 用户高频物品 | **不注入** | 避免高频偏见（用户不一定总买同样的东西） |
| 历史修正记录 | 仅命中的 alias | 未命中则传空列表，避免无关 alias 干扰 |
| 品类库 | 按 embedding 相似度取 top 10 | 从全量品类中筛选最相关的，而非全量注入 |
| 知识库经验 | 召回结果 | 仅 `user_corrected` 来源的记录才展示"曾被误认为"标注 |

### 2.5 Refine Prompt 防锚定设计

`build_refine_prompt()` 只传入物品的**外观描述**（如"白色利乐砖包装"），**不传入 Stage 1 的猜测名**（如"牛奶"），避免锚定效应。Stage 2 基于视觉描述 + RAG 上下文独立判断。

### 2.6 替换安全门控

Stage 2 结果不会无条件覆盖 Stage 1：

| Stage 2 结果 | 处理 |
|-------------|------|
| 名字不同 且 置信度 ≥ Stage 1 | 采信替换 |
| 名字不同 且 置信度 < Stage 1 | **拒绝替换**，保留 Stage 1 结果 |
| 名字相同 且 置信度更高 | 更新置信度和描述 |
| 名字相同 且 置信度不更高 | 保留原结果 |

---

## 用户修正流程

Stage 2 完成后，用户可以：

1. **修正物品名称** → 标记 `is_corrected=True`，触发 Gemini 重新描述正确物品外观
2. **修正数量**
3. **修正操作方向**（放入 ↔ 取出）
4. **补录遗漏物品** → 标记 `is_corrected=True`

---

## 后置处理：知识库写入 + 品类匹配

### 知识库写入门控

`_correction_flow()` 中，写入 `item_knowledge` 表前需通过三道门控：

| 条件 | 是否写入 | 说明 |
|------|---------|------|
| 用户修正过（`is_corrected=True`） | 写入 | 最可信来源 |
| 置信度 ≥ 0.7 且 Stage1=Stage2（名字一致） | 写入 | 两个独立信号一致 |
| Stage1 ≠ Stage2 且未修正 | **拒绝** | 有争议的结果不污染知识库 |
| 置信度 < 0.7 且未修正 | **拒绝** | 不可靠结果不进入知识库 |

### "误识别"标注保护

embedding 文本中的"该物品极易被误识别为 X"标注**仅在 `is_corrected=True` 时才写入**。防止 Stage 2 的自动改名结果把正确答案标记成错误。

### 双轨 Embedding 生成

每个物品生成两套 embedding（一次批量 API 调用）：

1. **知识库向量**：包含易错线索（用户修正时才加），用于高精度召回
2. **品类归类向量**：纯净版（仅名称+描述），避免语义被易错线索污染

### 品类匹配

`rag/category_matcher.py` → `match_category()`：

```
输入：物品名称 + 描述 + 纯净版 embedding
         │
         ▼
┌─ Step 1: Embedding 相似度匹配 ─────────────────────┐
│  与 item_categories 表中所有品类的 embedding          │
│  批量计算余弦相似度（numpy 向量化）                    │
│  阈值 ≥ 0.75 → 命中，归入该品类                      │
└─────────────────────────────────────────────────────┘
         │ 未命中
         ▼
┌─ Step 2: Gemini 文本推理 ──────────────────────────┐
│  物品名+描述+现有品类列表发给 Gemini                  │
│  判断属于哪个现有品类，或建议新建                      │
└─────────────────────────────────────────────────────┘
         │
         ▼
┌─ Step 3: 结果处理 ─────────────────────────────────┐
│  匹配已有品类 → 更新品类的 user_names 列表            │
│  新建品类 → 生成 embedding，写入 item_categories 表   │
└─────────────────────────────────────────────────────┘
```

```
视频帧 → 门状态检测 → 片段切割 → 关键帧提取
                                      │
                                      ▼
                        ┌─ Stage 1：盲猜 ──────────────┐
                        │ 关键帧图片 → Gemini（无上下文）  │

                                ▼ 逐物品处理

                │  品类库筛选（embedding top 10）    │

                        │
                        ▼
                ┌─ Stage 2：定向验证 ──────────────┐
                │ 物品专属帧 + 动态 RAG 上下文       │

                └───────┬──────────────────────────┘
                        │

                ┌─ 用户修正 ──────────────────────┐

                └───────┬────────────────────────┘
                        │
                        ▼
                ┌─ 知识库写入（三道门控）─────────┐
                │ 用户修正 → 写入                  │

                └───────┬────────────────────────┘

```

## Embedding 技术细节

- **模型**：`gemini-embedding-001`（Gemini embedding API）
- **存储**：embedding 向量序列化为 JSON 字符串，存储在 SQLite TEXT 列中
- **计算**：使用 numpy 批量矩阵运算（`batch_cosine_similarity`），一次计算查询向量与所有候选的相似度
- **批量生成**：一次分析会话中的所有物品共享一次 embedding API 调用（`get_embeddings_batch`）

## 修正记录的聚合加权机制

历史修正记录不是简单的"原始名 → 修正名"一对一映射，而是**按正确物品名聚合**的加权结构。

### 聚合逻辑

`rag/user_history.py` → `get_alias_mappings()` 将同一物品的所有修正记录聚合为一条：

```
item_knowledge 表中的多条记录：
  source=user_corrected, original_name="无糖可乐", item_name="培根", description="..."
  source=user_corrected, original_name="猪肉",     item_name="培根", description="..."
  source=user_corrected, original_name="火腿肠",   item_name="培根", description="..."

聚合为：
  {
    "corrected": "培根",
    "wrong_names": ["无糖可乐", "猪肉", "火腿肠"],
    "correction_count": 3,
    "description": "扁平长方形透明塑料盒，红白相间肉片"
  }
```

### 分级注入策略

`rag/prompt_enhancer.py` → `build_context()` 根据修正次数分级注入：

- **修正 ≥ 2 次**：加重警告语气 + 附带外观特征描述，引导 VLM 格外注意
- **修正 1 次**：简短提示 + 可选外观描述

修正次数越多的物品排序越靠前，VLM 在 prompt 中越早看到，注意力权重越高。

### 描述来源

外观描述来自用户纠错后 Gemini 的重新描述（`gemini_client.redescribe_item()`），存储在 `item_knowledge.description` 字段。高频物品模块（`get_frequent_items`）同样优先使用纠错后的描述，确保 VLM 看到的是经过用户验证的准确描述。

## 知识库去重与上限机制

`item_knowledge` 表用于积累物品知识，同一物品可能从不同角度/场景产生多条不同描述。为防止长期膨胀，采用**去重 + 上限封顶**策略。

### 去重规则

`storage/inventory.py` → `save_to_knowledge()` 写入前检查：

1. **描述完全相同** → 跳过，不重复写入。若新记录是 `user_corrected` 而已有记录是 `vlm_accepted`，则升级 source 优先级
2. **描述不同** → 正常写入（保留不同角度的描述信息）

### 上限封顶

同一 `user_id + item_name` 最多保留 **5 条**记录（`MAX_KNOWLEDGE_PER_ITEM = 5`）。超出时按以下优先级淘汰：

1. 优先淘汰最旧的 `vlm_accepted` 记录
2. `user_corrected` 记录优先保留（用户修正的信息价值更高）
3. 若 `vlm_accepted` 不够淘汰，则从最旧的记录中补齐

---

## 品类库初始化

首次运行时（`rag/category_init.py`），调用 Gemini 生成 50-80 个常见食物品类及其外观描述，批量生成 embedding 并写入 `item_categories` 表。后续新物品匹配不到时自动扩展品类库。

---

## 与传统 RAG 的区别

本项目借鉴了 RAG 的核心思想（检索外部知识增强生成），但实现方式是 **Context Injection** 而非 query-based retrieval：

| 维度 | 传统 RAG (Query-based Retrieval) | 本项目 (Context Injection) |
|------|---------|--------|
| 触发方式 | 根据用户 query 动态检索 | 每次分析前固定注入用户历史上下文 |
| 检索对象 | 非结构化文档片段 | 结构化数据库记录 |
| 检索方式 | 向量相似度检索 | SQL 查询 + 向量匹配（知识库/品类库） |
| 检索时机 | 生成前检索 | Stage 2 生成前（动态上下文注入）+ 生成后（品类匹配） |
| 增强方式 | 检索结果拼入 prompt | 结构化上下文注入 prompt |
| 知识积累 | 通常静态 | 每次运行后动态积累用户修正和物品知识 |
| 质量控制 | 通常无 | 三道门控防止不可靠结果污染知识库 |

## Notes
为什么不使用传统多模态（图片-描述对应）rag？

冰箱内摄像头角度固定且视野受限 — 物品经常被手遮挡、被其他物品挡住、只露出侧面或底部。同一瓶牛奶，正面放和侧面放在图片特征上差异巨大。

物品入场时的视觉状态极不稳定 — 用户手持物品放入/取出时，物品在运动中、有运动模糊、光照变化大（冰箱灯 + 外部光线交替），这些帧提取的图片特征噪声很高。

同品类外观高度相似 — 两盒不同品牌的牛奶在冰箱视角下几乎一样（都是白色纸盒），图片特征很难区分。但文本维度上"伊利"和"蒙牛"是完全不同的。

当前的文本 embedding + RAG 上下文注入已经够用 — 真正帮助识别准确率的是"这个用户上次放了什么"和"之前把白色纸盒修正为伊利纯牛奶"这类历史知识，这些用文本检索就能很好地解决。

当前的方案（让 VLM 直接看图识别 + 用文本 RAG 提供上下文引导）已经是针对这个场景最合理的架构。把精力花在图片特征检索上，投入产出比不高。
