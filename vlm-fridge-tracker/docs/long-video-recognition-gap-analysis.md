# 长视频多物品操作漏识别分析

> 场景：2分钟视频，用户连续拿取/放入数十个物品，Gemini 只能识别出一部分。

## 原因分析（按影响程度排序）

### 1. 门状态合并 — 一个片段塞进几十次操作（最关键）

`segment_extractor` 按"开门→关门"切割片段，`MERGE_GAP_FRAMES = 15`（约3秒）内的短暂关门会被合并。

如果用户2分钟内频繁拿取而没有完全关门再开门，整段视频会被识别为**一个巨大的 OPEN 片段**，所有操作塞进同一次 VLM 调用。

- 相关代码：`config.py:28` (`MERGE_GAP_FRAMES`)、`segment_extractor.py:54-64`（合并逻辑）
- 核心矛盾：当前架构是"一个片段 → 一次 VLM 调用"，长片段 = 信息量过载

### 2. VLM 对长图片序列的注意力衰减

即使发了 240 张关键帧给 Gemini（2分钟 × 2帧/秒），VLM 存在：

- **Lost-in-the-middle 问题**：序列中段的图片容易被模型忽略
- **输出 token 压力**：Prompt 要求逐帧报告 `hand_observations`，物品多时输出量巨大，模型可能提前截断或跳帧
- **相似物品合并倾向**：模型倾向于将外观相似的多次操作合并为一次

这是模型本身的能力限制，无法通过 Prompt 调优彻底解决。

### 3. Prompt "宁漏勿错" 策略

`prompt_template_v4.py:8`：
> "宁可漏报也不要误报。不确定时标注 confidence < 0.5"

快速连续拿取时画面模糊，模型在不确定时会直接不报告。这条规则在单次少量物品时合理，但在密集操作场景下会导致大量漏报。

### 4. 关键帧采样未必命中关键瞬间

`keyframe_selector.py` 按运动量加权采样，但：

- 运动量大 ≠ 正在拿放物品（可能只是手在翻找）
- 真正拿起/放下物品的瞬间可能只有1-2帧，被采样漏掉的概率不小
- `np.random.choice()` 无固定 seed，每次运行结果不同

### 5. 无漏检补救机制

当前的二次识别（refine）只对 **已识别但置信度低** 的物品做二次分析（`main.py:464-468`）。

对于 Gemini **完全没提到的物品**，没有任何补救手段——漏了就是漏了。

## 可能的改进方向（待讨论）

### 方向 A：长片段二次拆分

将一个长 OPEN 片段按"运动量峰值"或"固定时间窗口"进一步拆分成子片段，每个子片段独立调用 VLM，最后合并去重。

- 优点：直接解决"信息量过载"的根本问题
- 代价：VLM 调用次数增加，API 成本上升
- 关键难点：子片段间的去重逻辑（同一物品可能跨子片段边界）

### 方向 B：滑动窗口 + 增量识别

用固定大小的帧窗口滑动扫描长片段，每个窗口调用一次 VLM，将上一个窗口的识别结果作为上下文传给下一个窗口。

- 优点：自然处理时序关系，不会漏掉窗口边界的操作
- 代价：串行调用，延迟高；上下文传递的 Prompt 设计复杂

### 方向 C：调整 Prompt 策略

- 去掉或弱化"宁漏勿错"规则，改为"尽量报告所有观察到的手部交互，不确定时标低置信度"
- 简化 `hand_observations` 的逐帧报告要求，减少输出 token 压力
- 分两阶段：先让 VLM 粗扫整段视频列出所有可能的事件时间点，再针对每个时间点精细识别

### 方向 D：前置运动分析辅助

在调用 VLM 之前，用 CV 手段（光流、运动量峰值检测）先定位"手部活跃区间"，只把活跃区间的帧发给 VLM，大幅减少噪声帧。

### 方向 E：漏检补救

分析 VLM 返回的 `fridge_state_after`（操作后冰箱内物品列表），与操作前的状态对比，自动发现差异物品并触发定向二次识别。

## 评估建议

以上方向不互斥，可以组合使用。建议优先级：

1. **A（长片段拆分）** — 改动集中在 `segment_extractor` 和 `main.py`，效果最直接
2. **D（前置运动分析）** — 减少噪声帧，提升每次 VLM 调用的信息密度
3. **C（Prompt 调整）** — 改动最小，但效果有限
4. **E（漏检补救）** — 作为兜底机制
5. **B（滑动窗口）** — 实现复杂度最高，可作为长期方案
